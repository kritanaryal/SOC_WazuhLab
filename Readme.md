I built this lab because I was tired of just reading about SIEMs, threat hunting, and incident response in textbooks. I wanted to get my hands dirty, break things, and actually see what an attack looks like from the defender's side.

## 🧠 What I Actually Built

The goal wasn't just to install software; it was to simulate a real enterprise environment. 
* **The Brain:** An Ubuntu server running the Wazuh Manager, Indexer, and Dashboard.
* **The Victims (Agents):** A couple of VMs (Windows and Linux) that I intentionally attack to generate telemetry.
* **The Intelligence:** VirusTotal API integration to automatically check suspicious files.

## ⚠️ The Reality Check: What You Actually Need

When I first tried building this, I severely underestimated how resource-hungry a SIEM is. The Wazuh Indexer (which is basically Elasticsearch) will silently crash if it runs out of memory. **I made the mistake of not giving the server enough memory at first. Make sure you assign at least 8GB of RAM, or the database will crash on you.**

### The Hardware (My VM Setup)
* **The SIEM Server (Ubuntu 22.04 LTS):** 
  * You need at least **4 CPU Cores** and **8GB of RAM** (honestly, give it 12GB if you have it). 
  * Give it at least **50GB of storage**. Logs eat up disk space ridiculously fast.
* **The Endpoints (Windows 10 / Ubuntu):** 
  * 1 Core and 2GB RAM is fine for these. They just sit there and send logs.

### The Network
I assigned a static IP to my Wazuh Server so the agents wouldn't lose connection if my router rebooted. You also need to open a few ports on the server:
* `443` (So we can see the web dashboard)
* `1514` (The pipeline where agents send their logs)
* `1515` (The door agents knock on to register for the first time)

## 🛠️ Preparing the Server

Before touching Wazuh, the Ubuntu server needs to be ready. This is exactly what I ran on my fresh install.

```bash
# 1. Update everything. Never start a project without doing this.
sudo apt update && sudo apt upgrade -y

# 2. Install prerequisite packages:
sudo apt install curl apt-transport-https unzip wget libcap2-bin software-properties-common lsb-release gnupg2 -y

# 3. Configure Time Synchronization (NTP): Accurate time is vital for SIEM log correlation.
sudo apt install chrony -y
sudo systemctl enable --now chrony

# 4. Set up UFW (Uncomplicated Firewall):
sudo ufw allow 22/tcp
sudo ufw allow 443/tcp
sudo ufw allow 1514/tcp
sudo ufw allow 1515/tcp
sudo ufw enable

```
## 🚀 Wazuh Installation 
For lab environments, the Wazuh Installation Assistant is the most efficient and error-free method to deploy the complete stack (Indexer, Server, and Dashboard).

 ***1. Deploying the Central Server***
Execute the quickstart script from the official Wazuh repository:
```bash
curl -sO [https://packages.wazuh.com/4.8/wazuh-install.sh](https://packages.wazuh.com/4.8/wazuh-install.sh)
sudo bash ./wazuh-install.sh -a
#Note: This process takes 10-15 minutes. Once complete, the script will output the URL, Username (admin), and an autogenerated(random
)Password.
```
**2. Accessing the Dashboard**  
Navigate to ```https://<WAZUH_SERVER_IP>``` in your web browser. Accept the self-signed certificate warning (expected in a lab) and log in with the provided credentials.

**3. Deploying Wazuh Agents**  
To collect telemetry, deploy agents to your endpoint VMs.   
i. In the Wazuh Dashboard, click Wazuh > Agents > Deploy New Agent.  
ii. Select the target OS (e.g., Windows, Debian).  
iii. Enter the Wazuh Server IP address.   
iv. Copy the generated deployment command and run it in an elevated command prompt/terminal on the target endpoint. 
  
Example for Linux Agent:  
```bash
wget [https://packages.wazuh.com/4.x/apt/pool/main/w/wazuh-agent/wazuh-agent_4.8.0-1_amd64.deb](https://packages.wazuh.com/4.x/apt/pool/main/w/wazuh-agent/wazuh-agent_4.8.0-1_amd64.deb)
sudo WAZUH_MANAGER='192.168.1.50' WAZUH_AGENT_NAME='Linux-Endpoint-01' dpkg -i ./wazuh-agent_4.8.0-1_amd64.deb
sudo systemctl daemon-reload
sudo systemctl enable wazuh-agent
sudo systemctl start wazuh-agent
```
## 🔗 Integration Steps  
Integrating external threat intelligence and response tools drastically enhances the SOC lab.  
  
**VirusTotal Integration (Malware Detection)**    
To automatically scan file hashes detected by Wazuh FIM against VirusTotal:  
i. Obtain a free API key from VirusTotal.  
ii. Edit the ossec.conf file on the Wazuh Manager:  
```bash
sudo nano /var/ossec/etc/ossec.conf
```
iii. Add the following block within the ```<ossec_config>``` section:  
   **XML**
```
<integration>
  <name>virustotal</name>
  <api_key>YOUR_VIRUSTOTAL_API_KEY</api_key>
  <rule_id>554, 550</rule_id>
  <alert_format>json</alert_format>
</integration>
```
iv. Restart the Wazuh Manager: ```sudo systemctl restart wazuh-manager```.  
  
## ⚙️ Configuration and Tuning
Out-of-the-box SIEMs generate noise. Tuning is required to prioritize high-fidelity alerts.  

Enabling Vulnerability Detection  
By default, vulnerability detection may be disabled. To enable it:  
i. Open ```/var/ossec/etc/ossec.conf``` on the manager.  

ii. Locate the ```<vulnerability-detector>``` block.  

iii. Set ```<enabled>yes</enabled>```.  

iv. Enable the OS-specific feeds (e.g., Canonical, RedHat, Windows) by setting their respective ```<enabled>``` tags to ```yes```.

v. Restart the manager.    

## Custom Rules (local_rules.xml)
To suppress a false positive or elevate a specific alert, create a custom rule in ```/var/ossec/etc/rules/local_rules.xml```.  
  
  Example: Elevate failed SSH logins for a specific user (e.g., 'root') to Level 12:
  **XML**  
  ```bash
<group name="syslog,sshd,">
  <rule id="100001" level="12">
    <if_sid>5716</if_sid>
    <match>root</match>
    <description>SSHD: High-priority failed login attempt for root user.</description>
    <mitre>
      <id>T1110</id>
    </mitre>
  </rule>
</group>
```

## ✅ Testing and Validation  

Verify your SOC lab is actively ingesting and processing logs.  

i. Verify Agent Connection: Run ```sudo /var/ossec/bin/agent_control -l``` on the manager to see a list of Active agents.

ii. Trigger an SSH Brute Force Alert (Linux): From an attacker VM, attempt to SSH into a Linux agent with incorrect credentials 5-10 times. Look for Alert Rule 5712 in the Wazuh Dashboard.

iii. Trigger a File Integrity Alert (Windows/Linux):
Modify a monitored file. (e.g., ```sudo nano /etc/passwd``` or add a dummy/fake user). The dashboard should reflect a FIM modification alert within a few minutes.  

## Testing the Defenses

Building the lab is great, but I wanted to prove that it actually detects attacks. Here is how I tested it to make sure everything worked:

**1. The SSH Brute Force Test**
I used a Kali Linux VM to try logging into my Ubuntu agent with the wrong password multiple times on purpose. Within seconds, Wazuh noticed the pattern of failed logins and triggered a Level 10 Brute Force alert on my dashboard. 

**2. The Malware Test**
To test malware detection, I created a standard EICAR file (a safe, fake virus file used for testing) on my Windows desktop. Wazuh noticed the new file right away, checked it against VirusTotal, and successfully flagged it as malicious.  

## 🩹 Troubleshooting (Things I Messed Up) 
If you are building this, you will probably hit these same walls:

i. Dashboard says "Server not ready": Your VM ran out of RAM and the Indexer crashed. Give the VM more memory and run ```sudo systemctl restart wazuh-indexer```.  
ii. Agent is installed but says "Disconnected": I forgot to open port 1514 on my server's firewall. The agent was trying to send logs, but the server was dropping them.  
iii. I changed the config, but nothing happened: You must run ```sudo systemctl restart wazuh-manager``` every time you touch a .xml or .conf file.
